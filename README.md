## Current Remote-ADB Direction

The practical bootstrap path is:

```text
laptop USB ADB
  -> enable phone adbd TCP on 127.0.0.1:5555 / device port 5555
  -> configure the APK with VM SSH tunnel credentials
  -> APK opens an outbound SSH reverse tunnel to the VM
  -> VM sees the phone ADB service on VM localhost
  -> VM runs adb connect 127.0.0.1:<remote-port>
```

ADB is only the first endpoint behind the tunnel. Once the VM has ADB, the coding agent can install/update the APK, push additional tools, inspect logs, tap UI, and add more tunnel endpoints as needed.

The APK implementation uses a pure-Java SSH client so the shipped app does not need a native `ssh` binary per CPU architecture. The older laptop bootstrap script can still push a temporary SSH client as a fallback while the APK tunnel is being proven.

## Nontechnical Install Model

For app-level pairing without USB, the clean flow is a VM-specific download link:

```text
https://<vm>/download/pucky.apk?pair=<short-lived-token>
```

The token is generated by the VM, expires quickly, and is consumed once. If the user installs from that link, the APK can pair to the VM automatically after install/open. QR codes or manual pairing codes are fallback flows for generic APK downloads, not the primary user path.

Full ADB power still needs one trusted bootstrap step on stock Android:

- USB ADB once, preferred for development.
- Manual Android Wireless Debugging pairing, more technical.
- Root/system/device-owner paths, not the current plan.
